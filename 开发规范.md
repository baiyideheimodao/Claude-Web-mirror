# 多 AI 协作代码规范 – Claude 官网复刻项目

## 技术栈（固定）

- 框架：Vue 3.3+（Composition API，`<script setup>` 语法）
- 构建工具：Vite 5+
- 语言：TypeScript 5+（严格模式）
- 样式：TailwindCSS 3+（基于已提取的 Claude 官网设计令牌）
- 状态管理：Pinia
- 路由：Vue Router 4（history 模式）
- HTTP 请求：axios（已封装）
- 代码规范：ESLint + Prettier（项目根目录已配置）

### 0.3 禁止事项（红线）

## 2. 组件开发规范

### 2.1 组件文件命名

- 通用组件：`PascalCase.vue`（如 `Button.vue`）
- 页面组件：`PascalCasePage.vue`（如 `HomePage.vue`）
- 模块内私有组件：`PascalCase.vue`

### 2.2 组件模板结构（强制顺序）

```vue
<script setup lang="ts">
// 1. Vue 基础导入
import { ref, computed, onMounted } from 'vue'

// 2. 类型导入（type 关键字）
import type { SomeType } from '@/types'

// 3. 组件导入
import Button from '@/components/Button.vue'

// 4. 组合式函数导入
import { useDarkMode } from '@/composables/useDarkMode'

// 5. Props 定义
interface Props {
  title: string
  disabled?: boolean
}
const props = withDefaults(defineProps<Props>(), {
  disabled: false,
})

// 6. Emits 定义
const emit = defineEmits<{
  (e: 'click', value: string): void
}>()

// 7. 响应式数据
const count = ref(0)

// 8. 计算属性
const doubleCount = computed(() => count.value * 2)

// 9. 方法
function handleClick() {
  emit('click', 'hello')
}

// 10. 生命周期
onMounted(() => {
  // 初始化逻辑
})
</script>

<template>
  <div class="component-root">
    <Button :disabled="disabled" @click="handleClick">
      {{ title }}
    </Button>
  </div>
</template>

<style scoped>
/* 仅在 Tailwind 无法满足时使用 scoped 样式 */
</style>
```

### 2.3 Props 规范

- 必须使用 TypeScript 定义 `interface Props`
- 必须设置默认值（使用 `withDefaults` 或可选链）
- 禁止使用 `any` 类型
- 复杂类型使用 `PropType` 或单独定义 `type`

### 2.4 Emits 规范

- 必须使用 TypeScript 定义事件类型
- 事件名使用 kebab-case（模板中）或 camelCase（JS 中均可，推荐 camelCase）

## 3. 组合式函数（Composables）规范

### 3.1 命名

- 以 `use` 开头，camelCase（如 `useProductList`）
- 文件名与函数名一致

### 3.2 结构模板

```typescript
// composables/useCounter.ts
import { ref, computed } from 'vue'

export function useCounter(initialValue = 0) {
  const count = ref(initialValue)
  const double = computed(() => count.value * 2)

  function increment() {
    count.value++
  }

  function decrement() {
    count.value--
  }

  return { count, double, increment, decrement }
}
```

### 3.3 返回值

- 仅返回必要的状态和方法（避免暴露内部实现）
- 推荐返回对象（便于解构）

## 4. 样式规范（基于 TailwindCSS + 设计令牌）

### 4.1 优先使用 Tailwind 原子类

- ✅ 正确：`<div class="bg-brand-orange text-white p-4 rounded-lg">`
- ❌ 错误：`<div style="background: #C15F3C; ...">`

### 4.2 自定义样式必须使用设计令牌

- 所有品牌色、间距、字体等必须引用 `design-tokens.css` 中的 CSS 变量或 Tailwind 配置
- 禁止硬编码颜色值或尺寸

```css
/* design-tokens.css 示例 */
:root {
  /* 品牌色 - 使用 HSL 格式以支持 TailwindCSS 动态计算 */
  --brand-orange-hue: 15;
  --brand-orange-sat: 54.2%;
  --brand-orange-light: 51.2%;
  --brand-orange: hsl(var(--brand-orange-hue) var(--brand-orange-sat) var(--brand-orange-light));
  --brand-orange-dark: hsl(var(--brand-orange-hue) var(--brand-orange-sat) 40%);
  
  /* 字体 - 使用 Claude 官方字体栈 */
  --font-sans: "Anthropic Sans", system-ui, "Segoe UI", Roboto, Helvetica, Arial, sans-serif;
  --font-serif: "Anthropic Serif", Georgia, "Times New Roman", serif;
  --font-mono: "Anthropic Mono", ui-monospace, monospace;
  
  /* 间距 - 基于 4px 网格系统 */
  --spacing-md: 1rem;  /* 16px */
}
```

### 4.3 组件样式隔离

- 组件级样式必须使用 `<style scoped>`
- 全局样式仅放在 `src/styles/` 中，由架构师统一管理

## 5. 状态管理规范（Pinia）

### 5.1 Store 定义模板

```typescript
// stores/useUserStore.ts
import { defineStore } from 'pinia'

interface UserState {
  name: string
  isLoggedIn: boolean
}

export const useUserStore = defineStore('user', {
  state: (): UserState => ({
    name: '',
    isLoggedIn: false,
  }),
  getters: {
    displayName: (state) => state.name || 'Guest',
  },
  actions: {
    login(name: string) {
      this.name = name
      this.isLoggedIn = true
    },
    logout() {
      this.name = ''
      this.isLoggedIn = false
    },
  },
})
```

### 5.2 使用规则

- 模块私有状态放在模块内的 `stores/` 中
- 全局共享状态放在 `src/stores/global/` 中
- 禁止在组件中直接修改 state（必须通过 actions）

const request = axios.create({

## 9. 协作集成规则

### 9.1 分支模拟（物理隔离）

- 每个 AI 负责一个独立模块（如 `modules/home`、`modules/chat`）
- 两个 AI **不得同时修改同一个文件**
- 公共文件（`App.vue`、`router/index.ts`）由集成者（你）手动合并

### 9.2 增量交付

- AI 输出代码时，必须明确是“新增文件”还是“修改文件”
- 修改文件时，提供具体的 `diff` 或指出修改的函数/行号

### 9.3 依赖管理

- 禁止引入新的 npm 依赖，除非向架构师提出并更新 `package.json`

- ❌ 禁止在模块外修改其他模块的代码
- ❌ 禁止使用 `any` 类型（使用 `unknown` + 类型守卫）

## Git 提交信息格式规范

## 格式模板

```
<type>(<scope>): <subject>
```

[optional body]

[optional footer]

## Type 类型（必填）

- feat: 新功能
- fix: Bug 修复
- docs: 文档更新
- style: 代码格式（不影响功能）
- refactor: 重构（不是新功能也不是修复）
- perf: 性能优化
- test: 测试相关
- chore: 构建/工具配置
- ci: CI 配置
- revert: 回滚提交

## Scope 范围（可选但推荐）

常用 scope：

- home - 首页模块
- chat - 聊天模块
- components - 通用组件
- composables - 组合式函数
- stores - 状态管理
- router - 路由配置
- styles - 样式/设计令牌
- utils - 工具函数
- types - 类型定义
- deps - 依赖更新
- config - 配置文件

## Subject 主题（必填）

- 使用中文或英文均可，项目内统一（推荐中文）
- 不超过 50 个字符
- 首字母小写（英文）或正常中文句式
- 结尾不加句号
- 使用祈使句（如“添加”而不是“添加了”）

正确示例：
feat(home): 添加响应式布局支持
fix(chat): 修复滚动条位置错误
docs(readme): 更新环境配置说明

错误示例：
feat: 添加了新功能。（结尾句号，过去时）
fix bug（缺少 scope，描述不清晰）
feat(home): 这是一个非常非常非常非常非常非常长的主题（超长）

## Body 正文（可选）

- 详细描述为什么做这个改动和如何实现
- 每行不超过 72 字符
- 可以多行

示例：
feat(home): 添加 Hero 区域组件

实现首页 Hero 区域，包含大标题、副标题和 CTA 按钮。
使用 TailwindCSS 实现响应式布局，移动端字体自动缩放。
参考 Claude 官网设计令牌配置颜色和间距。

## Footer 脚注（可选）

- 关联 Issue 或标记 Breaking Change
- 关闭 Issue：Closes #123, #456
- 破坏性变更：BREAKING CHANGE: 描述

示例：
feat(api): 修改用户信息接口返回结构

将原来的扁平结构改为嵌套结构，以适配后端调整。

BREAKING CHANGE: 旧版本前端需同步修改 user store 中的取值路径。
Closes #789

feat(home): 实现特性卡片组件

添加三个特性卡片，展示 Claude 的核心能力。
卡片采用网格布局，移动端自动堆叠。

Closes #42
